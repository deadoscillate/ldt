# Adding a New Component Type

Sapio Forge components are part of the compiled rendering layer, not the source-of-truth logic model.

Add a new component only when it improves reusable rendering patterns across courses.

## 1. Define the component contract

Add the component type to `src/lib/course/types.ts`:

- add the string literal to `SCENE_COMPONENT_TYPES`
- add a typed interface for its props
- add it to `CompiledSceneComponent`

Keep the props constrained and declarative.
Do not add arbitrary script hooks.
If the component belongs to a simulation shell, decide which constrained slot it
is allowed to use (`header`, `main`, `sidebar`, or `footer`) instead of creating
freeform placement rules.

## 2. Validate the component schema

Update `src/lib/course/scenes.ts`:

- add a strict schema for the component
- include it in the discriminated union
- update any scene validation rules if needed

Validation should fail clearly for missing or invalid props.

## 3. Decide how it is produced

Choose how the component enters the render pipeline:

- derived from legacy node/layout fields during compile
- emitted by a future structured source field
- generated by a higher-level authoring helper

Do not bypass canonical compilation.

## 4. Register the React preview renderer

Update the scene registry in `src/components/scene/registry.tsx`:

- add the component renderer
- keep it presentational
- use the render context only for text interpolation and current runtime state

Do not embed branching logic in the component renderer.

## 5. Register the exported SCORM runtime renderer

Update the DOM registry in `src/lib/scorm/runtime-template.ts`.

The exported runtime should stay aligned with the preview renderer.

If the preview gains a new component but the SCORM runtime does not, output parity drifts.

## 6. Expose the change in Builder or inspector UI if useful

If the component is meant to be authorable now:

- surface its props in Builder view
- or expose it through the scene/component inspector

If it is only an internal compatibility component, document it and keep it out of the authoring UI.

## 7. Add tests

Minimum coverage:

- schema validation
- registry resolution
- rendered output
- legacy normalization if applicable
- SCORM/runtime parity where practical

## Rule of thumb

Add components to extend reusable visual patterns.

Do not add one-off render branches that belong only to a single course.
