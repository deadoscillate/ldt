# LDT Engine

LDT Engine is a structured authoring system for interactive training modules.

It is designed to feel closer to training-as-code than to a slide editor:

- source definitions live in plain-text YAML
- reusable templates and variables generate repeatable course variants
- browser preview is compiled from validated source
- SCORM packages are build artifacts, not the editable source of truth

The core product promise remains:

> Write branching training modules in YAML and export them as SCORM packages.

## What the product does

- provides a public landing page at `/`
- provides a studio at `/studio` for authoring, compiling, previewing, and exporting
- supports guided builder mode and raw YAML source mode
- validates course structure before preview or export
- compiles reusable templates and variables into a normalized runtime graph
- previews learner flow in the browser
- exports SCORM 1.2 packages
- includes waitlist, feedback, and lightweight beta intake tooling

## Routes

- `/`: landing page
- `/studio`: structured authoring studio
- `/admin`: minimal beta intake review page
- `/api/waitlist`: early-access capture
- `/api/feedback`: qualitative feedback capture
- `/api/events`: lightweight funnel events
- `/api/admin/intake`: JSON or CSV export for waitlist, feedback, and events

## Structured authoring model

The system is intentionally narrow.

- `course.yaml` or template YAML is the source of truth
- theme values, branching rules, templates, and variables live in source
- the browser preview is compiled from validated source
- the exported SCORM zip is a build output

This is not a PowerPoint replacement and not a drag-and-drop slide builder.

If you want the full model and workflow explanation, see:

- [docs/structured-authoring.md](docs/structured-authoring.md)
- [docs/lms-validation.md](docs/lms-validation.md)

## Studio workflow

The studio supports two complementary authoring modes.

### Builder view

Builder view is the beginner-friendly layer.

Use forms to:

- create steps
- choose node types
- assign layouts
- edit theme values
- configure branching and scoring
- preview immediately

The builder writes structured YAML internally. It does not replace the source model.

### Source view

Source view is the raw YAML editor.

Use it to:

- inspect the source generated by builder mode
- edit advanced fields directly
- upload existing YAML
- download source YAML for Git-backed workflows

### Compile workflow

The studio keeps the workflow explicit:

1. Edit source definition or builder fields.
2. Resolve template variables.
3. Validate schema, references, and layouts.
4. Compile the preview runtime.
5. Export a SCORM 1.2 build.

## Supported authoring model

### Node types

- `content`
- `choice`
- `branch`
- `quiz`
- `question`
- `result`

Author-facing aliases:

- `branch` compiles to the branching runtime node shape used by `choice`
- `question` compiles to the scored runtime node shape used by `quiz`

### Layout primitives

Each node can choose a presentation primitive:

- `title`
- `text`
- `image`
- `video`
- `two-column`
- `image-left`
- `image-right`
- `quote`
- `callout`
- `question`
- `result`

These layout values live in source, compile deterministically, and render consistently in both browser preview and exported SCORM runtime.

### Theme

Course source can include:

```yaml
theme:
  primary: "#1f6feb"
  secondary: "#f5f5f5"
  font: Inter
  logo: https://example.com/logo.svg
  background: "#ffffff"
```

Theme values are compiled into preview and SCORM runtime output.

## Reusable templates and variables

Templates support:

- base YAML structure
- reusable inline `blocks`
- compile-time `templateData`
- duplication into local source variants

Example:

```yaml
templateData:
  companyName: Example Corp
  introLine: Review the reporting workflow.

blocks:
  intro_block:
    - id: intro
      type: content
      title: "{{companyName}} reporting basics"
      body:
        - "{{introLine}}"
      next: question-1
```

Interpolation rules:

- only string placeholders are interpolated
- interpolation is compile-time only
- missing placeholders produce readable validation errors
- no scripting, `eval`, or arbitrary code execution is allowed

Runtime learner tokens like `{{score}}` and `{{maxScore}}` still work on result copy.

## Git-friendly project structure

The repo now includes a source layout designed for version control:

```text
/courses
  /phishing-awareness-baseline
    course.yaml
    template-data.yaml
    README.md

/templates
  /phishing-awareness-template
    template.yaml
    schema.yaml
    README.md

/assets
/docs
```

Recommended practice:

- keep `course.yaml` and `template-data.yaml` in Git
- review branching changes as text diffs
- version template updates separately from course-variant updates
- treat exported SCORM zips as generated artifacts

The studio supports:

- upload source YAML
- download source YAML
- import template-data YAML
- download template-data YAML
- duplicate a template into a local source variant
- download a small source-project README for Git-backed handoff
